## Agent Role Office

- Use specialist role lenses when a task benefits from domain-specific
  judgment, such as product ownership, C#/.NET backend, frontend, UI/UX design,
  visual art, QA, DevOps/release, security, documentation, or tech lead review.
- Select the smallest useful role set for the current task, identify the lead
  role when the work is broad, and synthesize specialist input into one
  accountable plan, implementation, or review result.
- Treat roles as professional scopes, not separate personalities. Do not create
  theatrical multi-agent dialogue unless the user explicitly asks for that
  brainstorming format.
- Let project-local stack, design system, business rules, runbooks, and memory
  override the default role catalog.
- Prefer narrow roles when they improve quality or speed. For example, route
  C#/.NET work through backend concerns such as async/await, cancellation,
  dependency injection, configuration, data access, API contracts, logging, and
  runtime reliability; route frontend work through framework conventions,
  routing, state, forms, accessibility, visual verification, and performance.
- Add new reusable roles gradually only when repeated work shows a real
  specialty gap. Do not promote one-off preferences, temporary incidents,
  product names, file names, or local examples into permanent shared roles.
- For broad delivery tasks, combine product-owner, tech-lead, stack-engineer,
  QA, and release perspectives as needed while keeping the actual changes
  scoped and verified.
- When a development effort begins, infer the best lead role or smallest role
  set from the project goal, stack inventory, README/runbook, project memory,
  manifests, and requested work after the initial context load. Briefly propose
  it before the first implementation plan when it improves direction or review
  quality.
- If the role choice is obvious and low-risk, state it as an operating
  assumption and continue. Ask only when different role choices would materially
  change scope, architecture, external systems, cost, data safety, or
  user-visible behavior.
- Revisit the role only after a meaningful project pivot, not on every minor
  task in the same thread.
- Follow `patterns/AGENT_ROLE_OFFICE.md` for the full role catalog and
  coordination model.
