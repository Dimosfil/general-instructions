# Make Instruction Kit Metadata Recording Explicit

## Purpose

Prevent consuming projects from marking instruction-kit migrations as applied
before the migration file changes have actually been applied.

## Source Files

- `templates/check-instruction-kit-updates.template.ps1`
- `patterns/INSTRUCTION_KIT_MIGRATIONS.md`
- `COMMANDS.md`

## Project Files To Update

- `tools/check-instruction-kit-updates.ps1`
- `tools/project-memory/instruction-kit.json`
- local instruction files named by pending migrations

## Merge Rules

- Update the project-local update-check script so the default mode lists pending
  migrations only.
- Disable or remove metadata-only `-Apply` behavior. If kept for compatibility,
  make `-Apply` exit with an explanation instead of writing
  `tools/project-memory/instruction-kit.json`.
- Add an explicit `-RecordApplied` mode that records pending migrations only
  after an agent has applied migration file changes and verified them.
- Do not update `instruction_kit_version`, `applied_migrations`, or
  `last_update_check_at` before the migration's file changes exist locally.
- If metadata was already advanced too early, compare the local files against
  the migration requirements, apply missing file changes, then correct metadata.

## Verification

- Run the project update-check script without flags and confirm it only lists
  pending migrations.
- Run the script with `-Apply` and confirm it refuses to record metadata.
- After applying file changes, run the script with `-RecordApplied` and confirm
  metadata changes only then.
- Reread changed instruction files.
- Run `git diff --check` when the project is a git repository.

## Rollback Or Conflict Notes

If a project has custom automation that already safely applies file changes
before recording metadata, preserve that behavior and add the explicit
post-apply metadata-recording guard.
