# Migration: Auto-Register Gateway FTP Projects

Migration ID: `2026.07.04.6__auto_register_gateway_ftp_projects`

## Purpose

Make `gi ftp <deploy-hub-path>` work as a project-to-gateway deploy handoff for
new projects when the gateway defines deterministic project registration. The
project agent should let the gateway derive the target from the source project,
not ask the user to choose a remote folder or fall back to a root target.

## Apply

- When a deploy gateway supports automatic project registration, use the current
  project root folder name as the default project id unless the gateway contract
  defines another id source.
- Let the gateway derive public URL, remote path, deploy mode, provisioning, and
  target naming from its own documented convention.
- Register or update the project in the gateway-owned deploy registry before
  upload when the project is not mapped yet.
- Record non-secret metadata needed for later hub, card, or index maintenance,
  such as project id, public URL or target name, source identity, deploy artifact
  path, deploy time/status, and pending-card state.
- If the gateway manages a separate public hub or site catalog, leave a pending
  record in the gateway's documented inbox or registry so the gateway agent can
  later compare new projects and update that catalog.
- Do not ask the user for remote folder, subdomain, or project id when the
  gateway contract already defines deterministic registration from the source
  project.
- Do not fall back to a gateway root/default remote path for an unknown project.
  If registration, provisioning, or artifact selection is missing or fails, stop
  and report the gateway blocker instead of uploading the whole source
  repository or guessing a destination.

## Gateway Contract

Deploy gateways that want this behavior should document:

- the project id source, usually the source root folder name;
- the target naming convention;
- the deploy registry or target map to update;
- the pending hub/card/index record location;
- the artifact selection rule or required build output;
- the provisioning and verification steps for first deploy.
