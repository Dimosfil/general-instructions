# Add Coherent Batch Verification

Migration ID: `2026.06.21.9__add_coherent_batch_verification`

## Summary

Add a reusable rule for finishing meaningful implementation, refactor,
migration, and configuration cleanup batches with source-of-truth consistency,
durable project-memory writeback, scoped diffs, and evidence-backed checks.

## Accepted Guidance

- Treat a coherent batch as one reviewable set of changes tied to one product,
  architecture, configuration, or verification goal.
- After meaningful batches, check every touched layer that can carry the same
  default, policy, workflow, contract, state, or interpretation rule.
- Prefer one authoritative source for changeable values and behavior contracts;
  other layers should load, render, validate, adapt, or generate from it.
- If duplication is intentional, document the sync or generation path and
  verify it.
- Update durable project-memory specifications for meaningful behavior,
  workflow, data-model, integration, or architecture changes. Do not use
  `pending-tasks.md`, task-manager notes, chat summaries, or commit messages as
  substitutes for durable specs.
- Inspect the changed-file list before finishing and remove or explain
  unrelated edits, generated noise, local-only artifacts, and accidental
  formatting churn.
- Treat `git diff --check` as a whitespace/error check, not as a substitute for
  tests. Distinguish harmless line-ending warnings from actual whitespace
  errors.
- Report remaining risks at the behavior or architecture level, not only as file
  names.

## Verification

- Run targeted searches for the changed default, policy, workflow, or contract
  across backend, frontend, tests, docs, build metadata, and project memory.
- Run the fastest relevant tests or smoke checks for the touched area.
- Run `git diff --check` and classify any output.
- Inspect `git diff --stat` or the changed-file list for unrelated files.

## Rollback

Remove only the coherent batch verification pattern, metadata, and references.
Do not remove underlying configuration-boundary, project-memory, code-quality,
or `gi refactor` rules.
