# Migration: Add Feature Workflow Contracts

Version: `2026.06.11.2`

Apply this migration to copied instruction kits that need durable product
workflow agreements for complex features.

## Changes

- Add guidance that project-specific feature behavior agreements should be
  recorded as feature workflow contracts.
- Add or update local project instructions so agents read relevant workflow
  contracts before changing a feature and preserve user-visible guarantees
  unless the user explicitly changes the agreement.
- Add a project-local template such as
  `templates/FEATURE_WORKFLOW_CONTRACT.template.md` or
  `docs/features/FEATURE_WORKFLOW_CONTRACT.template.md` when the project keeps
  feature specs in docs.
- Use workflow wording rather than only pipeline wording, because feature
  behavior may branch through states, errors, retries, background work, and
  terminal outcomes.

## Suggested Project Rule

- When the user or team agrees that a feature must work in a specific sequence
  or state flow, record it as a project-local feature workflow contract.
- Before changing a feature with a recorded contract, read the contract and
  preserve its user-visible guarantees unless the user explicitly changes them.
- If implementation changes the agreed workflow, update the contract in the
  same scoped change and report the behavior change.
- Contracts should cover entry points, required order, branches, blocking versus
  background work, loading/empty/error states, cancellation/retry behavior, data
  freshness, observability, and verification.

- Run `git diff --check`.
- Confirm the project has a documented place for feature workflow contracts or

- Confirm no project-specific private workflow details are added to shared
