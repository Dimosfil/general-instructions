# Keep One-Off Probes Out Of Tools

Migration ID: `2026.07.13.2__keep_one_off_probes_out_of_tools`

## Summary

Make script lifecycle, reuse, and ownership the deciding factors for `tools/`
placement so single-task research and diagnostic executables cannot be
misclassified as durable tooling.

## Accepted Guidance

- Do not treat Python, PowerShell, shell, or any other executable file type as
  proof that a script belongs in `tools/`.
- Keep single-task research probes, exploratory scripts, one-source scrapers,
  ad hoc collectors, data-inspection programs, and throwaway diagnostics out of
  `tools/`, including new `tools/research`, `tools/probes`, and `tools/scratch`
  subtrees.
- Prefer an inline command. If a file is genuinely required, use the project's
  documented ignored scratch/temp location outside `tools/`, remove it after
  use, and preserve only necessary outputs in the documented evidence or
  artifact location.
- Promote a script into `tools/` only when it has a reusable project-owned
  purpose, stable invocation contract, documentation, verification, and an
  expected future caller. Being useful for the current task is not enough.
