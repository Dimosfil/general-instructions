# Abstract Concrete Examples Before Shared Rules

Migration ID: `2026.06.21.12__abstract_concrete_examples_before_shared_rules`

## Summary

Add guidance that concrete requests, bugs, screenshots, demos, and implementation
details are evidence for reusable rules, not the rule itself. Agents must extract
the portable principle before adding shared instructions or generic runtime
contracts.

## Scope

- `AGENTS.md`
- `CHANGELOG.md`
- `VERSION.md`
- `patterns/CONFIGURATION_BOUNDARIES.md`
- `patterns/DEVELOPMENT_TOOL_PRODUCT_BOUNDARIES.md`
- `templates/AGENTS.template.md`
- `tools/project-memory/instruction-kit.json`
- `templates/instruction-kit.template.json`

## Accepted Guidance

- When deriving a reusable rule from a concrete request, bug, screenshot, demo,
  or implementation detail, first name the portable principle.
- Do not promote incidental entities, object types, years, filenames, local
  paths, data partitions, model names, UI labels, query text, selected runs, or
  workflow examples into shared defaults, product policy, architecture, or
  examples.
- Treat the concrete case as evidence. Restate the rule in neutral terms before
  adding it to shared instructions.
- Put changeable selections behind configuration, manifests, contracts,
  adapters, task payloads, user-selected state, or project-local memory.
- If implementation advice starts naming the incidental case as the solution,
  stop and restate the generic rule before proposing project-specific changes.

## Verification

- Search added shared-rule text for incidental concrete entities, filenames,
  paths, years, model names, UI labels, queries, and selected-run names.
- Confirm any remaining concrete values are necessary, explicitly illustrative,
  and replaceable.
- Confirm the rule can be copied into an unrelated project without suggesting
  that the project depends on the incident that produced the rule.

## Rollback

Remove only this abstraction-before-shared-rules guidance and metadata. Do not
remove existing configuration-boundary, product-boundary, or project-agnostic
explanation rules.
