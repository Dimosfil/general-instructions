# Clarify WorkNest Next-Task Endpoint Contract

Migration ID: `2026.05.30.1__clarify_worknest_next_task_endpoint_contract`

## Purpose

Make the WorkNest `next-task` API contract explicit so agents do not guess the
wrong HTTP method or parameter shape during `gi start sprint` workflows.

## Apply

- Update the WorkNest adapter endpoint list to document
  `GET /agent-intake/next-task?project=<project>&sprintId=<sprintId>`.
- Before running a WorkNest sprint workflow, verify required endpoint methods
  and query parameters against the adapter contract, not just endpoint paths.
- If a task-manager endpoint returns an unexpected method, parameter, or routing
  error, re-read the adapter endpoint docs before trying any workaround.
- If the documented contract still does not match the running service, report a
  stale or misconfigured manager endpoint and stop before sending or completing
  work.

## Verify

- Confirm the WorkNest adapter shows the full `next-task` method and query
  parameter signature.
- Confirm the `gi start sprint` workflow tells agents to re-read adapter endpoint
  contracts before working around unexpected endpoint errors.
- Run `git diff --check`.

## Rollback Or Conflict Notes

This migration clarifies the currently documented WorkNest adapter contract. If
a project-local WorkNest API intentionally uses a different route shape, that
project's adapter docs must override this shared reference explicitly.
