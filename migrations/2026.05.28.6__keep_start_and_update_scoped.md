# Keep Start And Update Scoped

Migration ID: `2026.05.28.6__keep_start_and_update_scoped`

## Purpose

Prevent agents from turning lightweight GI startup/update commands into stale
plan continuation or broad Git synchronization.

## Shared Source Files

- `AGENTS.md`
- `COMMANDS.md`
- `templates/AGENTS.template.md`
- `migrations/2026.05.28.6__keep_start_and_update_scoped.md`

## Project Files

- `AGENTS.md`
- copied command reference, if present

## Apply

- For `gi start`, `gi restore`, and title-only startup messages, restore only
  compact orientation for the next turn.
- Mention remembered plans, stale task notes, old refactoring phases, or local
  commits ahead of a remote only as context when relevant.
- Do not offer to continue, run, finish, or push remembered work unless the user
  explicitly asks for that action.
- Keep `gi обновить` scoped to accepted instruction-kit updates and migrations.
- Do not reinterpret `gi обновить` as a request to push pre-existing local
  commits, sync a feature branch, resume a remembered plan, or perform general
  Git maintenance.
- Commit or push only changes created by the update flow itself and only when
  local update policy permits it.

## Verification

- Run `git diff --check`.
- In a project with old memory notes and local commits ahead of a remote,
  confirm `gi start` asks for the current task instead of proposing to continue
  or push old work.
- In a project with local commits ahead of a remote, confirm `gi обновить` does
  not ask to push those commits unless the update flow created them.

## Rollback Or Conflict Notes

If a project has a more specific local startup or update policy, preserve the
local policy and adapt only the scoping rule that prevents stale-plan or broad
Git-sync behavior.
