# Verify gi reboot startup success

## Summary

Tighten `gi reboot` / `gi restart` so agents cannot report success from a newly
created PID alone.

## Instructions

- Treat `gi reboot`, `ги ребут`, `gi restart`, and `ги рестарт` as requests to
  start or restart the current application using project-local run instructions.
- If the app is running, restart it; if it is not running, start it in the
  background so focus does not jump away from the user's current window.
- After launch, wait briefly and verify the documented startup success signal:
  a still-running expected process, visible desktop window when applicable,
  health/discovery endpoint for web/API apps, and relevant startup or crash logs
  when documented.
- Do not report reboot success from a PID alone.
- If the process exits, no expected window or health signal appears, or a new
  startup traceback is present, report the reboot as failed or unverified with
  the concrete evidence.

## Verify

- Confirm `gi reboot` still uses project-local run instructions.
- Confirm desktop apps are launched in the background.
- Confirm success reporting requires a post-launch startup signal beyond PID
  creation.

## Rollback Or Conflict Notes

If a project has no documented success signal, use the narrowest available
evidence, such as process survival after a short wait and absence of a new crash
log, and report the startup as unverified instead of successful.
