# Migration: Continue Artifact Upload For Pending Gateway Entry

Migration ID: `2026.07.04.11__continue_artifact_upload_for_pending_gateway_entry`

## Purpose

Clarify that a pending deploy-gateway inbox/request entry means devops or
hosting publication is waiting, not that the project agent should skip upload.
The project agent should still build and hand off the current artifact whenever
the gateway provides a documented pending, staging, queue, or handoff upload
target.

## Apply

- Treat an existing pending inbox/request entry as active deploy state that
  should continue through artifact upload when a documented pending/staging or
  handoff target exists.
- Keep domain, subdomain, hosting-slot, and public-card provisioning as the
  devops/hosting responsibility after artifact handoff.
- Stop only for an errored/rejected gateway request or when no documented
  upload/handoff target exists. The stop report must be an explicit deploy error
  with failed step, evidence, responsible system or owner, next required action,
  and artifact/source state already recorded.
- Continue to forbid fallback uploads to root/default paths, duplicate mappings,
  or user prompts for another remote folder when a pending gateway entry already
  identifies the project.

## Verify

- Confirm runtime deploy rules say pending domain/hosting requests do not by
  themselves stop artifact upload.
- Confirm `patterns/PROJECT_FTP_DEPLOY.md` mirrors the pending upload handoff
  behavior.
- Confirm `COMMANDS.md`, `VERSION.md`, instruction-kit metadata, and template
  metadata include this migration.

## Rollback Or Conflict Notes

This migration does not authorize guessing a remote folder. If the gateway has
no documented pending/staging/handoff upload target, report that as an explicit
deploy error instead of uploading to a default root.
