# Add GI Logic Command

Migration ID: `2026.07.07.2__add_gi_logic_command`

## Summary

Add `gi logic` / `ги логика` as a GI command for recovering current-project
logic or adopting portable logic from an explicitly named URL, repository, or
local folder.

## Instructions

- Treat `gi logic` / `ги логика` without a source as a request to inspect the
  current project only and build or update a durable logic map in project memory:
  core domain modules, workflow contracts, invariants, data flows, integration
  boundaries, evidence paths, and verification gaps.
- Treat `gi logic <source> [focus]` / `ги логика <source> [focus]` as explicit
  permission to inspect that source only for the scoped logic-adoption task.
  Keep the current project as the write target unless the user explicitly asks
  to modify the source project.
- When a focus term is supplied, such as a component, client, bot, worker, API,
  parser, or workflow name, search for that focus before broad scans.
- Read external sources narrowly: instructions, README/docs, manifests,
  project-memory specs, entry points, and focused source modules. Do not read
  secrets, private runtime data, logs, databases, user-home app data, generated
  artifacts, unrelated sibling repositories, or broad private folders.
- Extract portable behavior contracts and module responsibilities before code
  changes. Do not blindly copy another project's source, machine-specific config,
  generated output, or private data.
- If implementation is requested or clearly implied, adapt the logic into the
  current project's architecture and configuration boundaries, update project
  memory with the source/evidence mapping, and run the smallest documented
  checks that cover the adopted behavior.
