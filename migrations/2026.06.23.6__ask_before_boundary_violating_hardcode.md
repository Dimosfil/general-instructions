# Ask Before Boundary-Violating Hard-Code

Migration ID: `2026.06.23.6__ask_before_boundary_violating_hardcode`

## Purpose

Prevent agents from silently hard-coding changeable product, language, prompt,
synonym, intent, model-behavior, or compatibility values as a quick fix for
tests, legacy samples, or observed incidents.

## Apply

- Keep changeable product, language, prompt, synonym, intent,
  query-normalization, ranking, model-behavior, and operational-policy values in
  documented config, resource files, adapters, curated data assets,
  project-local memory, or user-selected state.
- If a quick fix, legacy compatibility path, test expectation, or observed
  sample would require hard-coding one of those values, first implement the
  compliant boundary.
- Ask one concise clarification question only when the correct source of truth,
  config location, or temporary compatibility layer is not documented.
- When asking, name the boundary conflict and the exact scoped exception or
  config choice needed. Do not use the question as permission to ignore the
  rule silently.

## Verify

- Search touched implementation paths for new inline product names, prompt
  strings, synonym dictionaries, translation maps, intent heuristics, query
  expansions, model compatibility hacks, and legacy sample aliases.
- Confirm any remaining compatibility behavior is isolated in a documented
  config/resource/adapter layer with validation and a removal or ownership path.
- Run the project's relevant config/schema checks, focused tests, or smoke
  checks.
