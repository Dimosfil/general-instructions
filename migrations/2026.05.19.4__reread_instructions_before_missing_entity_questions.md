# Reread Instructions Before Missing Entity Questions

## Purpose

Prevent agents from treating another project folder or the shared instruction
library as a runtime fallback when a local file, skill, config, endpoint, task,
or other entity is missing.

## Source Files

- `AGENTS.md`
- `COMMANDS.md`
- `patterns/FIRST_MESSAGE_HANDLING.md`
- `templates/AGENTS.template.md`
- `templates/AGENT_WORKING_AGREEMENTS.template.md`
- `templates/instruction-kit.template.json`

## Project Files To Update

- `AGENTS.md`
- `tools/AGENT_WORKING_AGREEMENTS.md`
- `tools/project-memory/instruction-kit.json`

## Merge Rules

- Add a general missing-entity fallback rule:
  - If a required file, skill, config, script, endpoint, task, or other entity
    is missing or not found, first reread the relevant local instructions,
    runbook, project memory, and accepted instruction-kit artifacts for the
    current scope.
  - If the entity is still missing, ask the user one short clarification
    question.
  - Do not use another project folder or the shared instruction library as a
    runtime fallback unless the user explicitly gives that path and action.
- For `gi` commands, keep the command in the current project root. The shared
  library remains a source for accepted update artifacts during bootstrap or
  update workflows, not a live dependency for missing runtime pieces.
- Do not weaken existing rules that allow documented API, connector, or
  task-manager endpoint access for cross-project communication.

## Verification

- Reread edited instruction files.
- Validate `tools/project-memory/instruction-kit.json` if it is updated.
- Run `git diff --check`.

## Rollback Or Conflict Notes

If a project intentionally keeps a skill or config outside the project root,
record that as explicit project-local guidance naming the path and allowed
action. Do not infer it from a missing-file error.
