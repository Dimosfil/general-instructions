# Add Task Manager Add Sprint Command

Migration ID: `2026.06.06.3__add_task_manager_add_sprint_command`

## Summary

Add a reusable `gi add sprint` / `gi create sprint` / `gi добавить спринт`
command for creating visible executable Sprints/Cycles through the configured
task manager.

## Scope

- `AGENTS.md`
- `COMMANDS.md`
- `patterns/SERVICE_DISCOVERY_CONFIG.md`
- `skills/task-manager-plans/SKILL.md`
- `skills/task-manager-plans/references/managers/worknest.md`

## Accepted Guidance

- Treat `gi add sprint`, `gi create sprint`, `gi добавить спринт`, and
  equivalent add-sprint wording as requests to create a visible executable
  Sprint/Cycle through the configured task manager.
- Resolve the manager service id through config-service, read the manager
  contract, and use only documented sprint/cycle creation or executable plan
  creation operations.
- Normalize the sprint title, goal, and task list into the adapter's documented
  executable shape. If the user did not provide sprint content and no current
  plan exists, ask for it.
- Read the created Sprint/Cycle back and verify lifecycle identifiers, URL,
  status, or task list when the manager supports those fields.
- Stop and report the exact blocker when auth, permissions, method, parameters,
  routing, schema, project, lifecycle, readback, or requested object type is
  missing or mismatched.
- Do not downgrade add-sprint requests to raw intake, Work Items, local
  checklists, one-task plans, or any other substitute object.
- For WorkNest, use the documented executable plan payload such as `type:
  "plan"`, `project`, `title`, and non-empty `items[]`, unless the live contract
  documents a newer equivalent. A raw receipt alone is not a created Sprint.
- For WorkNest, resolve `service_id: "worknest"` through config-service before
  every workflow. Historical local URLs such as `http://127.0.0.1:4190` are
  operator context, not canonical runtime config.
- WorkNest's external-agent API is `/agent-intake/...`: `GET /health`,
  `GET /agent-intake/contract`, `POST /agent-intake/raw`,
  `GET /agent-intake/next-task?project=<project>&sprintId=<sprintId>`, and
  `POST /agent-intake/task-completed`.
- WorkNest manager owns status transitions, timing, archival, ordering, and task
  movement unless the live contract explicitly exposes a lifecycle operation to
  the external agent.
- Markdown sprint files such as `sprint.md` and numbered task files are
  WorkNest's internal model. External agents use the API and must not edit those
  files directly unless they are running as local agents inside the WorkNest
  project under project-local instructions.
