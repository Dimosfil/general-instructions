# Require Task Manager Active Task Lifecycle

Migration ID: `2026.06.06.2__require_task_manager_active_task_lifecycle`

## Summary

Make configured task managers the source of truth for executable agent work.
Agents must get active or next tasks through the documented manager contract,
send progress, blocker, and completion notes back through the manager, and stop
when lifecycle operations or requested object types are unavailable.

## Scope

- `AGENTS.md`
- `COMMANDS.md`
- `patterns/SERVICE_DISCOVERY_CONFIG.md`
- `skills/task-manager-plans/SKILL.md`
- `skills/task-manager-plans/references/managers/worknest.md`

## Accepted Guidance

- Treat `gi active task`, `gi next task`, `gi get task`, and equivalent
  active-task wording as requests to obtain executable work from the configured
  task manager.
- Resolve the manager service id through config-service, read the manager
  contract, and use only documented active-task, next-task, start/progress,
  blocker, completion, and readback operations.
- Use only lifecycle identifiers returned by the manager. Do not infer task,
  sprint, cycle, or project IDs from raw intake receipts, UI text, file paths,
  old notes, or guessed slugs.
- Send task status, progress, blocker, and completion notes back to the manager
  when the manager supports those lifecycle operations.
- Stop and report the exact blocker when the contract, auth, permissions,
  lifecycle IDs, status update, readback, or requested object type is missing or
  mismatched.
- Do not create a raw intake receipt, local checklist note, one-task plan, Work
  Item, or other substitute object when the user requested a specific visible
  Cycle/Sprint/task object and the documented operation is blocked.
- For WorkNest, a `401 Authentication required` response from
  `/product/cycles` or the documented cycle operation means Cycle creation is
  blocked. The agent must report the UI-auth/API-permission blocker instead of
  creating a Work Item and claiming the Cycle exists.
