# Add RAG Retrieval Eval Checks

Accepted version: `2026.06.18.5`

## Rule

Treat `gi tools rebuild evals` and `gi rag rebuild evals` as requests to run the
project's configured RAG health and retrieval eval checks without rebuilding
source indexes unless the project-local command explicitly does so.

Prefer machine-checkable retrieval evals that verify:

- configured RAG files are readable and free of secrets;
- generated indexes are ignored when rebuildable;
- SQLite chunk counts, semantic corpus counts, and vector record counts match
  when those layers are enabled;
- exact keyword retrieval finds command names, paths, symbols, versions, and
  error text;
- semantic retrieval finds conceptual project guidance;
- hybrid retrieval can satisfy questions through keyword or vector evidence;
- expected source paths appear in top results.

Do not make the primary eval target a model's free-form answer wording. Test the
retrieval evidence first, then let normal source verification govern any answer
or code change.

## Rationale

An apparently successful vector rebuild only proves that records exist. A useful
RAG system also needs cheap checks that detect stale layers, count mismatches,
missing generated-ignore rules, and recurring retrieval misses.

## Propagation

Apply this rule to:

- live repository instructions;
- copied-project templates;
- RAG system structure and semantic retrieval guidance;
- project-memory README and local eval runner;
- instruction-kit metadata.
