# Split Project Documentation And Memory Layers

## Purpose

Separate user/project documentation from implementation-driving project memory.
Project documentation should describe the project's overview, functionality,
technology stack, commands, operations, and troubleshooting. Project memory
should preserve the algorithms, business logic, workflow contracts, state
machines, invariants, architecture decisions, and verification guarantees that
code must follow.

## Merge Rules

- Keep `README.md`, `docs/`, and runbooks as the project documentation layer for
  overview, visible functionality, stack, commands, operations, examples, and
  troubleshooting.
- Keep `tools/project-memory/` as the project memory layer for behavior that
  drives implementation: algorithms, business rules, workflows, state machines,
  invariants, architecture contracts, integration contracts, and verification.
- If an existing project stores stack facts at
  `tools/project-memory/specs/technology-stack.md`, keep that compatibility path
  or link it to a canonical docs file. Do not maintain two independent stack
  inventories.
- If user-facing functionality changed and only project memory was updated,
  add or update project documentation.
- If algorithms, business rules, states, integrations, or architecture contracts
  changed and only project documentation was updated, add or update project
  memory.
- Update `tools/project-memory/instruction-kit.json` only after the file changes
  have been applied and checked.

## Verification

- Confirm a project documentation layer exists through `README.md`, `docs/`, or
  the runbook.
- Confirm implementation-driving behavior is documented under
  `tools/project-memory/` when the project has non-trivial features, business
  rules, integrations, or architecture contracts.
- Confirm there is one canonical source for stack facts and operational
  commands, with links from compatibility paths if needed.
- Validate edited JSON instruction-kit metadata.
- Run `git diff --check` or the project equivalent when the project is a git
  repository.

## Rollback

Revert only the documentation-layer guidance and metadata entry. Do not delete
existing project documentation or project-memory specs.
