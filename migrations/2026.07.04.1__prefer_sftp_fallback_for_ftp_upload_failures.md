# Prefer SFTP Fallback For FTP Upload Failures

Migration ID: `2026.07.04.1__prefer_sftp_fallback_for_ftp_upload_failures`

## Summary

Clarify that failed, stalled, or timing-out FTP/FTPS uploads must move to the
authorized SFTP-over-SSH fallback when that route is documented, instead of
repeating the same FTP transfer path or bypassing FTPS certificate validation.

## Accepted Guidance

- Treat upload stalls, hangs, repeated timeouts, and failed stream opens as
  failed FTP/FTPS transfers.
- When FTP/FTPS upload fails or is unreliable, immediately inspect
  project-local FTP config, selected service contracts, and current
  user-provided deployment details for an authorized SFTP-over-SSH route to the
  same remote deploy folder.
- If the SSH host, port, user, credential reference, and remote folder are
  available, switch to SFTP over SSH before more FTP/FTPS upload variants and
  report that fallback.
- If required SFTP details are missing, report the exact missing fields or ask
  one short question for them. Do not invent SSH credentials, private-key paths,
  or remote paths.
- Do not disable TLS certificate validation or accept invalid FTPS certificates
  as a routine fallback unless the project-local deploy contract or current
  user message explicitly authorizes that degraded security path.
