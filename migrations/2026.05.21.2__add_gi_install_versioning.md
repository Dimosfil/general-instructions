# Add GI Install Versioning

Migration ID: `2026.05.21.2__add_gi_install_versioning`

## Purpose

Require `gi install` workflows to preserve project versioning in production
builds and installer artifacts.

## Shared Source Files

- `AGENTS.md`
- `COMMANDS.md`
- `VERSION.md`
- `CHANGELOG.md`

## Project Files

- `AGENTS.md`

- `tools/project-memory/instruction-kit.json`

## Apply

- Before packaging for `gi install`, resolve the application version from
  project-local metadata such as manifests, package files, assembly attributes,
  release files, or installer configs.
- Update the version in the production build, installer metadata, and installer
  filename or artifact name when the local tooling supports it.
- If the build, installer, or versioning contract is missing or ambiguous, ask
  one concise clarification question instead of inventing a version.
- Do not use shared-instruction version numbers such as `VERSION.md` as the
  application version for another project.

## Verify

- Ask `gi install` in a project with explicit version metadata; confirm the
  agent reports the version used and produces or identifies an installer
  artifact whose filename includes the same version when the local tooling
  supports versioned artifact names.
- Ask `gi install` where version metadata is split between app and installer
  configs; confirm the agent detects mismatches and resolves them using
  project-local instructions or asks one short clarification question.
- Ask `gi install` in a project with no clear versioning contract; confirm the
  agent does not invent a version.

## Rollback Or Conflict Notes

This migration only adds versioning discipline to installer builds. It must not
grant permission to publish releases, sign artifacts, upload installers, or
modify unrelated release metadata unless the user explicitly asks and the local
project instructions support that workflow.
