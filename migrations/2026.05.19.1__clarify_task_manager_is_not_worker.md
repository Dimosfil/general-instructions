# Clarify Task Manager Is Not The Worker

## Purpose

Prevent agents from describing a task manager as the actor doing project work.
The agent works the task; the task manager records, assigns, orders, and tracks
task state.

## Merge Rules

- When starting sprint work, say the agent takes the task in work through the
  manager, not that the manager takes the task in work.
- Treat manager status changes such as `in_progress` as task-list bookkeeping
  and assignment state, not proof that the manager is executing the task.
- Use task-manager APIs to request work, record progress, submit completion
  notes, and preserve the task list.
- Keep responsibility clear in reports: the agent implements and verifies; the
  manager stores the work queue and lifecycle metadata.

## Verification

- Reread task-manager rules and adapter docs.
- Confirm user-facing examples do not imply that the task manager performs the
  implementation work.
- Run `git diff --check` when the project is a git repository.

## Rollback Or Conflict Notes

If a project has an automated worker inside its task manager, document that
separately. Do not let that special case blur the default external-agent role.
