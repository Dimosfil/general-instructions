# Prioritize GI Start Sprint Manager Workflow

Migration ID: `2026.06.19.2__prioritize_gi_start_sprint_manager_workflow`

## Purpose

Make active-sprint commands route through the configured task manager instead
of being swallowed by the generic `gi start` startup-restore rule.

## Shared Source Files

- `AGENTS.md`
- `COMMANDS.md`
- `patterns/FIRST_MESSAGE_HANDLING.md`
- `patterns/RAG_STARTUP_FLOW.md`
- `patterns/SERVICE_DISCOVERY_CONFIG.md`
- `skills/task-manager-plans/SKILL.md`
- `templates/AGENTS.template.md`

## Apply

- Treat `gi start sprint`, `gi sprint start`, and equivalent active-sprint
  wording as more specific than plain `gi start`.
- Restore only the startup context required for task-manager work, then resolve
  the configured manager through config-service.
- Read the manager guide when present, then read the manager contract.
- Request the active Sprint/Cycle or next task through the documented manager
  operation.
- Move work through the manager's documented lifecycle states and submit
  completion through the manager contract.
- Stop with the exact blocker if discovery, auth, contract, lifecycle,
  completion, or object-type support is missing.
- Do not fall back to generic `gi start`, local task notes, raw intake, guessed
  endpoints, or filesystem task edits.
