# Forbid Cross-Project Filesystem Access

## Purpose

Keep each agent scoped to its current project root. Agents must not browse,
edit, or infer state from other project folders just because a task manager,
summary, or previous chat mentions them.

## Source Files

- `AGENTS.md`
- `patterns/FIRST_MESSAGE_HANDLING.md`
- `skills/task-manager-plans/SKILL.md`
- `skills/task-manager-plans/references/managers/worknest.md`
- `templates/AGENTS.template.md`
- `templates/AGENT_WORKING_AGREEMENTS.template.md`
- `templates/instruction-kit.template.json`

## Project Files To Update

- `AGENTS.md`
- `tools/AGENT_WORKING_AGREEMENTS.md`
- project-local task-manager adapter notes or runbooks

## Merge Rules

- Treat the current project root as the filesystem boundary for normal work.
- Do not read, search, edit, create, delete, move, or inspect files in another
  project or arbitrary external folder.
- Communicate with external project systems through their documented APIs,
  connectors, or task-manager endpoints.
- Use filesystem access outside the current project only when the user gives an
  explicit, concrete instruction naming the target path and action, such as
  "write this file to D:\path\folder".
- Task-manager paths, raw intake metadata, sprint paths, summaries, logs, or
  historical chat context are not permission to enter another project folder.
- If completing a task appears to require another project's files, stop and ask
  for explicit permission or for an API/connector route.

## Verification

- Reread scope and task-manager rules.
- Confirm sprint/task-manager guidance does not imply agents may inspect
  manager-owned project folders directly.
- Run `git diff --check` when the project is a git repository.

## Rollback Or Conflict Notes

If a project intentionally uses a monorepo or configured multi-root workspace,
document the allowed roots in that project's own instructions. The default rule
remains current-root only.
