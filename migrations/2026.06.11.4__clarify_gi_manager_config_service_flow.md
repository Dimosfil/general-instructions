# Clarify GI Manager Config-Service Flow

## Summary

Make `gi manager`, `gi tm`, `gi manager test`, `ги менеджер`, and `ги манагер`
explicit task-manager inspection commands that resolve the configured manager
through GI config-service before using any task-manager API.

## Rationale

Agents may otherwise stop after seeing missing local endpoint details, use a
legacy `base_url`, or infer a task-manager runtime URL from stale memory. The
shared rule must route manager status/test commands through the same
config-service discovery flow used by active-task and sprint workflows.

## Apply

- Update `AGENTS.md` from the current shared template or add an equivalent rule:
  `gi manager`/`ги менеджер` commands read the enabled manager id or
  `service_id` from project-local task-manager config, resolve it through
  `GET /services/{serviceId}`, read `endpoints.guide` when present, read
  `endpoints.contract`, and use `endpoints.api` only for documented manager
  operations.
- Update `patterns/SERVICE_DISCOVERY_CONFIG.md` if copied locally so the task
  manager workflow lists `ги менеджер` and `ги манагер`, accepts `service_id`,
  and ignores legacy `base_url` values unless a local migration rule explicitly
  converts them into config-service records.
- Update any project-local `tools/project-memory/task-managers.json` that still
  stores a runtime `base_url`: keep only the enabled manager id or
  `service_id`, plus non-secret preferences such as workspace, project, and
  intake mode.

## Verification

- Validate edited JSON files with `ConvertFrom-Json`.
- Run `git diff --check`.
- For a configured manager, query config-service with
  `GET /services/{serviceId}` and verify the returned service record exposes
  `endpoints.contract` and `endpoints.api`.
