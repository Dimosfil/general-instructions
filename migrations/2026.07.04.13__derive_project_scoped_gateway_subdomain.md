# Migration: Derive Project-Scoped Gateway Subdomain

Migration ID: `2026.07.04.13__derive_project_scoped_gateway_subdomain`

## Purpose

Make automatic deploy-gateway registration choose a project-scoped public
target for unmapped projects. A gateway may have a shared base domain or public
hub, but a new project must not deploy to that apex/root domain by default.

## Apply

- For an unmapped source project, use the current project folder name as the
  default project id unless the gateway contract names another id source.
- Let the gateway derive the public URL, remote path, deploy mode, and
  provisioning requirements from its documented naming convention.
- Unless an explicit existing target mapping says otherwise, the derived public
  URL must be project-scoped, normally a sanitized project id under the
  configured base domain.
- Do not target the apex/root domain, shared hub hostname, or another project's
  hostname for an unmapped project.
- Do not ask the user to choose a remote folder, project id, or subdomain when a
  deterministic project-id-to-hostname rule exists. If that rule is missing,
  report the missing gateway contract instead of guessing.
- Keep root/default remote path and whole-repository upload prohibitions in
  force.

## Verify

- Confirm runtime deploy rules require project-scoped hostnames for unmapped
  gateway registrations.
- Confirm `patterns/PROJECT_FTP_DEPLOY.md` forbids apex/root domain fallback.
- Confirm `COMMANDS.md`, `VERSION.md`, instruction-kit metadata, and template
  metadata include this migration.

## Rollback Or Conflict Notes

This migration does not forbid an explicit existing apex-domain mapping for a
project that truly owns the apex domain. It forbids using the apex/root domain
as an implicit fallback for a new or unmapped source project.
