# Add Portable Project Memory Specifications

## Purpose

Propagate the accepted rule that project memory is the durable, portable product
specification layer, while `tools/summary/` is only compact chat handoff state.

Projects should record feature algorithms, business logic, workflow contracts,
data rules, architecture migration history, verification guarantees, and current
implementation maps in project memory so another agent can rebuild the same
behavior on a different language, framework, platform, or UI stack.

This migration also adds SQL/vector diagnostic rules and default activation
limits for generated project-memory retrieval stores.

## Merge Rules

- Add a clear summary-vs-project-memory distinction near existing startup,
  memory, or RAG rules.
- Add the rule that non-trivial feature, business-rule, data-model,
  integration, or architecture work must update the relevant project-memory
  specification in the same scoped change.
- Create `tools/project-memory/architecture-migrations.md` from the template if
  it does not exist. Preserve existing architecture history if the file already
  exists.
- Keep project-memory specs split by meaning, such as feature specs,
  business-rule docs, data-model docs, integration contracts, and architecture
  migrations.
- In `rag-system.json`, add recursive project-memory source globs and
  `activation_limits` for SQLite and vector retrieval without overwriting
  project-specific source groups, exclusions, collection names, or secret
  handling.
- Add `gi sql` / `gi sqlite` and `gi vector` as diagnostic commands. They report
  current counts, readiness, staleness, and recommendations; they do not deploy
  external services, install heavy dependencies, upload data, or index private
  sources by default.

## Verification

- Confirm the project instructions distinguish handoff summaries from durable
  project-memory specifications.
- Confirm feature/business/architecture work has a same-change project-memory
  writeback rule.
- Confirm `architecture-migrations.md` exists or the project has an equivalent
  architecture history file.
- Confirm `rag-system.json`, when present, contains SQLite and vector activation
  limits or stricter project-local equivalents.
- Confirm generated SQLite databases, semantic corpora, and vector indexes
  remain ignored when rebuildable.

## Rollback

Remove only the added portable-specification, architecture-history, activation
limit, and `gi sql` / `gi vector` diagnostic guidance. Do not delete existing
project-specific memory documents.
