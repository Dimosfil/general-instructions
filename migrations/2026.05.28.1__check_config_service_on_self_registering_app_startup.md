# Check Config Service On Self-Registering App Startup

Migration ID: `2026.05.28.1__check_config_service_on_self_registering_app_startup`

## Purpose

Make config-service registration resilient to changed config-service settings.
Applications that publish their own service records should not rely only on
cached or previously bundled discovery config when they start.

## Apply

- For applications that must register themselves in config-service, require a
  live config-service config check on every process startup before publishing or
  refreshing the app's own service record.
- Read the current `configServiceUrl` through the documented project-local
  override or GI main config before registration.
- Use cached or bundled config only as an explicit degraded-startup fallback
  documented by local run instructions.
- Do not hardcode another config-service address in startup scripts, generated
  service records, task-manager config, summaries, or agent memory.

## Verify

- Confirm self-registering app startup code or run instructions reads current
  config-service config before registration.
- Confirm the service record is published or refreshed against the current
  config-service URL.
- Confirm any cached-config fallback is explicitly documented as degraded
  startup behavior.

## Rollback Or Conflict Notes

If a project cannot contact config-service during startup, document that as a
local degraded-startup mode instead of silently registering against stale config.
