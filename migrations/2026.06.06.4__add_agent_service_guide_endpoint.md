# Add Agent Service Guide Endpoint

Migration ID: `2026.06.06.4__add_agent_service_guide_endpoint`

## Summary

Require agent-facing HTTP services to publish a compact service-owned guide
endpoint alongside their strict contract endpoint when they expose workflows to
external agents.

## Accepted Guidance

- Agent-facing HTTP services should expose a guide plus a contract, preferably
  `GET /agent/guide` and `GET /agent/contract`, or adapter-specific equivalents
  such as `GET /agent-intake/guide` and `GET /agent-intake/contract`.
- Config-service service records should publish `endpoints.guide` when the
  service exposes an agent guide, plus `endpoints.contract` and `endpoints.api`.
- Agents must resolve runtime URLs through config-service by `service_id`.
- Agents should read `endpoints.guide` first when present, then
  `endpoints.contract` before sending tasks, plans, or state-changing requests.
- Treat the guide as onboarding: startup steps, allowed actions, forbidden
  actions, ownership boundaries, required capabilities, safe payload examples,
  error policy, privacy policy, and version fields.
- Treat the contract as strict workflow validation: schemas, methods,
  lifecycle identifiers, capabilities, error shapes, and supported
  state-changing operations.
- Stop and report a mismatch when the guide and contract disagree about
  required endpoints, workflow ownership, lifecycle identifiers, permissions, or
  supported object types.
- Do not infer permissions from filesystem paths, stale memory, old dashboard
  URLs, raw intake receipts, local notes, or previous task-manager records.
- Guide and contract responses must not expose secrets, cookies, raw private
  payload dumps, production data, large logs, credentials, tokens, or private
  keys.
