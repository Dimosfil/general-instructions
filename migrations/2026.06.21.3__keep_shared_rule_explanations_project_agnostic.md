# Keep Shared Rule Explanations Project Agnostic

Migration ID: `2026.06.21.3__keep_shared_rule_explanations_project_agnostic`

Apply this migration to copied instruction kits so agents do not explain
general GI rules by anchoring them to one current repository, recent bug, demo,
product, or generated artifact.

## Project Files To Update

- Project-local `AGENTS.md`
- Project-local authoring rules, working agreements, or architecture guidance
  that describe how agents should write reusable instructions
- Shared-rule patterns that are often triggered by a concrete incident

## Rule To Add

- When explaining, documenting, or adding a shared GI rule, keep the explanation
  project-agnostic.
- Do not anchor a reusable rule in the current project, a recent bug, one demo,
  one product name, or one repository unless the user explicitly asks for that
  concrete comparison.
- Use neutral terms such as "a development tool", "a generated product",
  "a selected run", "a service", "a task manager", or "a workflow UI".
- If an example is necessary, mark it as illustrative and keep it replaceable.
- Do not let the explanation of a reusable rule reintroduce the same coupling
  the rule is meant to prevent.

## Verification

- Search new shared-rule text for concrete project names, product names, demo
  domains, customer names, repository names, and local paths.
- Confirm any remaining concrete names are either required by the user's request
  or explicitly marked as non-normative examples.
- Confirm the rule can be copied into an unrelated project without suggesting
  that project depends on the incident that produced the rule.

## Rollback

Remove only the added project-agnostic explanation guidance. Do not remove the
underlying product-boundary or hard-code rules.
