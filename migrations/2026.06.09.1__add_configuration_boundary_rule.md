# Migration: Add Configuration Boundary Rule

Version: `2026.06.09.1`

Apply this migration to copied instruction kits so projects avoid hard-coded
deployment, user, runtime, machine, service, credential, path, feature-flag, and
operational-policy values.

## Changes

- Add the configuration-boundary rule to the project's `AGENTS.md` or equivalent
  agent entry point.
- Copy or reference `patterns/CONFIGURATION_BOUNDARIES.md` when the project
  keeps shared patterns locally.
- Update project setup/run/deploy guidance to document where config examples,
  environment variables, service discovery records, and secret references live.
- When applying this migration to an existing codebase, audit for hard-coded
  ports, URLs, hostnames, credentials, private paths, user names, feature
  toggles, limits, model names, deployment folders, and environment-specific
  switches.
- Refactor low-risk findings into existing project-local config, environment
  variables, or service discovery records. If a full refactor is too large,
  record the remaining cleanup as a project-memory task instead of silently
  leaving it untracked.
- Avoid embedding machine-specific absolute paths in source, shared
  instructions, or committed examples. For path values accepted from config,
  resolve and validate absolute paths at startup or I/O boundaries.

## Verification

- Run `git diff --check`.
- Validate any edited JSON, TOML, YAML, env examples, or config schemas with the
  project's normal parser or check command.
- Run the relevant config/schema tests, unit tests, or smoke checks for the
  touched runtime path.
- Confirm committed examples contain no real secrets, private hostnames, private
  folders, tokens, or machine-specific absolute paths.
- Confirm any deferred hard-code cleanup is recorded in the project's durable
  planning or memory location.
