# Migration: Check Deploy Gateway Inbox Before Mapping Block

Migration ID: `2026.07.04.9__check_deploy_gateway_inbox_before_mapping_block`

## Purpose

Make deploy-gateway publication state explicit when a new site/domain is already
queued but the final hosting-project map is not ready yet. Agents must not report
only "unknown deploy project" or ask the user for another mapping when the
gateway already has a pending or blocked inbox/request entry for the source
project.

## Apply

- Update deploy/FTP gateway rules so an unmapped project check includes the
  gateway registry plus documented project inboxes, pending queues, hub-card
  queues, and domain/hosting request lists.
- Treat an existing pending or blocked inbox/request entry as the active deploy
  state. Refresh allowed non-secret metadata, report the concrete hosting/domain
  blocker, and wait for the documented devops or hosting follow-up path.
- When the gateway documents a new-domain, subdomain, hosting-slot, or public
  card request workflow, create or refresh that request before stopping in
  pending/provisioning status.
- Forbid duplicate mappings, gateway-root uploads, or user prompts for another
  remote folder when deterministic registration or a pending inbox entry already
  identifies the project.

## Verify

- Confirm `patterns/AGENTS_RUNTIME/09-project-operation-commands.md` requires
  checking gateway inbox/request state before reporting unknown project or
  missing target mapping.
- Confirm `patterns/PROJECT_FTP_DEPLOY.md` mirrors the same FTP deploy behavior.
- Confirm `COMMANDS.md` explains the user-facing behavior for automatic gateway
  registration and pending hosting/domain requests.

## Rollback Or Conflict Notes

This migration does not grant consuming projects permission to edit private
gateway secrets or hosting credentials. It only requires agents to use documented
gateway-owned request state before declaring the deploy target unknown.
