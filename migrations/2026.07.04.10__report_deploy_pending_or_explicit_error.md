# Migration: Report Deploy Pending Or Explicit Error

Migration ID: `2026.07.04.10__report_deploy_pending_or_explicit_error`

## Purpose

Prevent deploy-gateway flows from ending in vague "blocked" states. A new
domain/site request should either remain in a documented pending/provisioning
state, or produce an explicit deploy error that explains exactly why automatic
progress cannot continue.

## Apply

- Replace generic deploy "blocker" language with a two-state outcome:
  documented pending/provisioning state, or explicit deploy error.
- When a gateway inbox/request entry already exists, treat `pending` and
  `errored` entries as active deploy state, refresh allowed non-secret metadata,
  and report the concrete pending or error reason.
- If registration, target provisioning, or artifact selection cannot continue,
  the error report must include the failed step, evidence, responsible system or
  owner, next required action, and artifact/source state already recorded.
- Forbid fallback uploads to a root/default path, duplicate mappings, or prompts
  for another remote folder when the gateway has deterministic registration or
  an active inbox/request entry.

## Verify

- Confirm runtime deploy rules no longer instruct agents to stop with a vague
  gateway blocker.
- Confirm `patterns/PROJECT_FTP_DEPLOY.md` requires pending/provisioning or an
  explicit error report.
- Confirm `COMMANDS.md`, `VERSION.md`, instruction-kit metadata, and template
  metadata include this migration.

## Rollback Or Conflict Notes

This migration does not require agents to bypass hosting limits or invent
credentials. It requires honest status: keep waiting work as pending, and report
external rejections as explicit errors with next actions.
