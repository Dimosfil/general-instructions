## Apply

- Keep API keys, access tokens, service-account keys, webhook secrets, signing
  secrets, and similar credentials out of source code, committed config, client
  bundles, public frontend environment variables, logs, traces, generated
  artifacts, chat responses, and project memory.
- Route browser, mobile, and packaged-app calls through a backend, server-side
  function, proxy, or token-exchange service instead of embedding long-lived
  provider credentials.
- Prefer separate credentials per person, runtime service, application,
  environment, or integration boundary.
- Keep development, staging, and production credentials separate.
- Store production secrets in managed secret stores such as a hosting platform
  secret manager, KMS, vault, cloud secret store, deployment secret store, or
  equivalent approved backend.
- Use provider-supported scoping, usage monitoring, rotation, and network
  restrictions such as IP allowlists where available.
- Rotate and revoke credentials immediately when they may have leaked or left
  the control of their intended owner.

## Verify

- Confirm committed config, examples, docs, generated artifacts, logs, traces,
  and project-memory files do not contain real secrets.
- Confirm client-side bundles and public frontend environment variables do not
  contain long-lived provider credentials.
- Confirm production deployments resolve secrets from the documented secret
  backend or secret reference.
- Confirm suspected leaked credentials have been revoked or rotated before the
  issue is reported resolved.
