# Migration: Save Selected Deploy Gateway

Migration ID: `2026.07.04.5__save_selected_deploy_gateway`

## Purpose

Let each project remember its selected DevOps/deploy gateway after the first
explicit deploy command, so later short commands can deploy through the same
gateway without repeating the path.

## Apply

- Treat `gi deploy <path>` / `ги деплой <path>` and `gi ftp <path>` /
  `ги фтп <path>` as both a deploy request and a selection of that path as the
  current project's deploy gateway.
- Store the selection in an ignored project-local file such as
  `tools/deploy/deploy-gateway.local.json`.
- Record only local selection metadata there: gateway path, entrypoint,
  source-path parameter, project id, deploy mode, and target name when the
  gateway contract defines them.
- Keep real secrets, credentials, private remote paths, and gateway runtime data
  in the gateway's own ignored config or secret store.
- When a later `gi deploy`, `ги деплой`, `gi ftp`, or `ги фтп` has no argument,
  read the saved gateway first and deploy the current project through it.
- If a bare `gi deploy` / `ги деплой` has no saved gateway, stop and ask the user
  to make the full first call such as `gi deploy <method-or-path>` /
  `ги деплой <способ-или-путь>`. Do not reinterpret it as `gi prod`, direct FTP
  upload, a build, or another deployment workflow.

## Ignore Rules

Add `tools/deploy/deploy-gateway.local.json` to project-local ignore rules when
the project adopts this migration.
