# Require Executable Single-Task Intake Contract

## Purpose

Prevent task-manager adapters from accepting single-task intake that cannot be
executed through the advertised task lifecycle.

## Source Files

- `skills/task-manager-plans/SKILL.md`
- `skills/task-manager-plans/references/managers/worknest.md`
- `templates/AGENTS.template.md`
- `templates/AGENT_WORKING_AGREEMENTS.template.md`
- `templates/instruction-kit.template.json`

## Project Files To Update

- `AGENTS.md`
- `tools/AGENT_WORKING_AGREEMENTS.md`
- `tools/project-memory/task-managers.json` if it documents accepted intake
  types
- project-local task-manager adapter notes or runbooks

## Merge Rules

- For any task-manager workflow that accepts a single-task payload, require one
  of these explicit contracts:
  - convert the payload into an executable sprint/task record and return the
    identifiers needed by lifecycle endpoints such as next-task, task-completed,
    archive, or close; or
  - reject the payload with a clear contract error that tells callers to send a
    supported executable payload such as `type: plan`; or
  - document the payload as intake-only and do not report it as executable work.
- Agents must not create a replacement one-task plan to work around a raw task
  that lacks execution identifiers.
- If an intake receipt lacks required execution identifiers for the requested
  workflow, report the task-manager contract gap and stop or ask for the correct
  endpoint behavior.
- Capability checks must verify that every accepted intake type can be advanced
  through the advertised lifecycle endpoints, not only that intake was stored.

## Verification

- For each enabled task-manager adapter, inspect its accepted payload types.
- Confirm accepted task payload responses include enough identifiers for the
  configured lifecycle commands, or that unsupported payload types are rejected
  or documented as intake-only.
- Run the fastest project documentation or syntax check.
- Reread changed instruction files.

## Rollback Or Conflict Notes

If a project intentionally treats `type: task` or another single-task payload as
raw intake only, preserve that behavior but document it clearly and make agents
avoid using that receipt as proof of executable task creation.
