# Preserve Summary Intent And Integration Mapping

Accepted version: `2026.06.18.4`

## Rule

For `gi summary` / `gi —Å–∞–º–º–∞—Ä–∏`, do not reduce architecture or research
conversations to a generic list of facts. If the user is evaluating an external
project, article, pattern, or tool as a possible integration target, the handoff
must explicitly preserve the user's integration intent and map external
concepts to current project components.

Include the answers to these questions when relevant:

- What concrete task or project direction was the user exploring?
- Was the discussion informational, or preparation for implementation?
- What external project, article, pattern, or tool is being considered?
- Which current local components map to that external pattern?
- What should a future agent not miss when continuing the work?
- Which conclusions are decisions, and which are still hypotheses?

## Rationale

Architecture and research threads often matter because of the user's direction,
not only because of the facts mentioned in the thread. A future agent can
correctly continue only when it understands why the user studied the external
concept and how it might fit the current project.

## Propagation

Apply this rule to:

- live repository instructions;
- copied-project templates;
- summary templates;
- RAG startup and project-memory guidance;
- instruction-kit metadata.
