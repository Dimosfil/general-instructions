# Migration: Attempt Gateway Provisioning After Stale Error

Migration ID: `2026.07.04.12__attempt_gateway_provisioning_after_stale_error`

## Purpose

Prevent deploy-gateway flows from stopping on stale or indirect hosting-limit
evidence. An old inbox error, screenshot, cached host quota check, or external
claim that provisioning may fail should warn the user, not replace a current
gateway provisioning attempt.

## Apply

- Treat stale or indirect hosting/provisioning errors as warning context.
- When the gateway contract provides a safe create, refresh, request, or
  provisioning attempt for a domain, subdomain, hosting slot, or public card,
  tell the user that provisioning may fail and run the attempt.
- Decide from the current attempt result whether to continue artifact upload,
  leave the request pending, or return an explicit deploy error.
- Stop without attempting only when the current gateway contract has no
  documented attempt/refresh path, or when the current attempt returns an
  explicit rejection.
- Keep fallback prohibitions in force: do not create duplicate mappings, ask for
  arbitrary remote folders, upload to gateway roots/default paths, or guess
  credentials.

## Verify

- Confirm runtime deploy rules require a current provisioning attempt after
  stale error evidence.
- Confirm `patterns/PROJECT_FTP_DEPLOY.md` mirrors the warning-then-attempt
  behavior.
- Confirm `COMMANDS.md`, `VERSION.md`, instruction-kit metadata, and template
  metadata include this migration.

## Rollback Or Conflict Notes

This migration does not bypass real host limits. If the current provisioning
attempt fails, report the fresh error with failed step, evidence, responsible
system or owner, next required action, and artifact/source state.
