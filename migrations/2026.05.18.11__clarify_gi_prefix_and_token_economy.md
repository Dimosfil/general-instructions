# Clarify GI Prefix And Token Economy

## Purpose

Clarify that `gi` is the only short command prefix for the copied
`general-instructions` instruction kit in each project, and that the kit exists
primarily for token economy and RAG-style context restoration.

## Source Files

- `patterns/FIRST_MESSAGE_HANDLING.md`
- `templates/AGENTS.template.md`
- `templates/AGENT_WORKING_AGREEMENTS.template.md`
- `GENERAL_DEVELOPMENT_PLAYBOOK.md`
- `COMMANDS.md`

## Project Files To Update

- `AGENTS.md`
- `tools/AGENT_WORKING_AGREEMENTS.md`
- `tools/project-memory/instruction-kit.json`

## Merge Rules

- Add or update a local rule saying short chat commands that start with `gi`
  are the command surface for the copied `general-instructions` instruction kit
  in the current project.
- State that `gi` is the only short prefix and must not be renamed to `GAI` or
  another alias.
- State that the instruction kit is a token-economy and RAG-startup layer:
  restore only task-relevant context from local instructions, handoff summaries,
  targeted searches, accepted migrations, and project memory instead of broad
  repository reads or large chat outputs.
- Preserve project-specific command details and any local wording that is more
  specific.
- Update `tools/project-memory/instruction-kit.json` after the file changes are
  applied successfully, not before.

## Verification

- Reread the changed local instruction files.
- Confirm no rule introduces `GAI` as a valid command prefix.
- Confirm the local startup/update guidance still points to accepted release
  artifacts and does not tell consuming projects to read `updates/`.
- Run `git diff --check` when the project is a git repository.

## Rollback Or Conflict Notes

If the local project already has a different meaning for `gi`, stop and ask the
user how to resolve the command conflict before changing command rules.
