# Verify Test Plan Command Contracts

Migration ID: `2026.06.16.5__verify_test_plan_command_contracts`

## Summary

Require agents to verify current project-local command contracts before
recommending or running test, smoke-check, API, UI, or CLI verification steps.
This prevents stale examples from old chats, screenshots, handoff summaries, or
task notes from turning into wrong ports, routes, flags, or payload fields.

## Scope

- `AGENTS.md`
- `COMMANDS.md`
- `patterns/PROJECT_TESTING_STRATEGY.md`
- `templates/AGENTS.template.md`
- `templates/AGENT_WORKING_AGREEMENTS.template.md`
- `tools/project-memory/instruction-kit.json`
- `templates/instruction-kit.template.json`
- `VERSION.md`
- `CHANGELOG.md`

## Accepted Guidance

- When creating or running a test, smoke-check, or verification plan, verify
  exact commands, CLI flags, ports, routes, health endpoints, request payload
  fields, and environment variables from current project-local instructions,
  runbooks, manifests, config entry points, or source code.
- Treat handoff summaries, task notes, screenshots, and old chat examples as
  status evidence, not authoritative command contracts.
- If a summary names a check but current project-local contracts disagree,
  follow the current contract and mention the mismatch briefly.
- Keep config-service startup guidance aligned in templates: web-facing
  applications must not guess ports or bind fallback ports when service
  discovery is required.

## Verification

- Ask for a project test plan where a handoff summary and current source code
  disagree about a port or payload field. Confirm the agent checks the current
  local contract before recommending commands.
- Run `git diff --check`.
