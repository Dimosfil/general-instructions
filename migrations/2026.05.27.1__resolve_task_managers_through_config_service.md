# Resolve Task Managers Through Config Service

Migration ID: `2026.05.27.1__resolve_task_managers_through_config_service`

## Purpose

Make GI config-service the source of truth for local task-manager runtime URLs.
Project task-manager config should keep only the selected manager name or
service id plus non-secret project preferences.

## Apply

- Treat `gi config service` and `ги конфиг сервис` as explicit aliases for the
  GI config-service bootstrap/discovery flow.
- Treat `gi config service url=<url>`, `ги конфиг сервис url=<url>`, and
  `ги конфиг сервис урл=<url>` as the canonical way to declare the
  config-service URL for the current environment.
- Save the URL to the shared `configServiceUrl` pointer or to the explicit
  project-local override, depending on local instructions.
- Tell local services to read this URL before registering themselves or reading
  discovery records.
- Validate the URL before saving it: require a full `http://` or `https://` URL
  and reject secrets, tokens, usernames, passwords, query strings, or fragments.
- Store `service_id` for each enabled task manager, normally the same value as
  the manager `id`.
- Remove task-manager runtime URL fields from project memory when the manager
  is registered in config-service.
- Resolve manager runtime details with `GET /services/{serviceId}`.
- Use `endpoints.availability` for availability checks.
- Read `endpoints.contract` before using manager-specific workflow operations.
- Use `endpoints.api` for task-manager operations after reading the contract.
- Stop with a concise blocker if the manager id is missing or config-service has
  no matching service record.
- Do not scan sibling project folders, guess ports, or copy URLs from old
  task-manager memory as a runtime fallback.

Example project config:

```json
{
  "version": 1,
  "managers": [
    {
      "id": "worknest",
      "enabled": true,
      "service_id": "worknest",
      "workspace": "",
      "project": "worknest-core",
      "intake_mode": "raw"
    }
  ]
}
```

## Verify

- Confirm enabled managers have `service_id`.
- Confirm project task-manager config no longer relies on `base_url` for local
  manager discovery.
- Confirm config-service has a matching service record for each enabled manager.
- Confirm the task-manager contract endpoint documents the operation needed for
  the current workflow.

## Rollback Or Conflict Notes

If a project cannot use config-service yet, keep the old endpoint config only as
an explicit documented project-local override. Do not silently fall back to old
URLs when config-service is expected to be available.
