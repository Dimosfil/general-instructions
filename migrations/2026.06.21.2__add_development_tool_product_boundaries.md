# Add Development Tool Product Boundaries

Migration ID: `2026.06.21.2__add_development_tool_product_boundaries`

Apply this migration to copied instruction kits so developer tools,
orchestrators, task managers, agent harnesses, generators, scaffolding systems,
and workflow UIs do not bind themselves to one generated product, demo, task, or
customer domain.

## Project Files To Update

- Project-local `AGENTS.md`
- Project-local architecture or runbook docs for orchestrators, generators,
  task managers, workflow UIs, or agent harnesses
- `tools/project-memory/architecture-migrations.md` when the project had to
  remove an existing product-specific runtime coupling
- Relevant feature workflow contracts or implementation plans when workflow-log
  behavior changes

## Rule To Add

- Keep developer tools, orchestrators, task managers, agent harnesses, and code
  generators separate from the products they build. Do not hard-code one demo,
  customer, project type, selected run, product name, UI label, folder slug,
  stack, or task contract as a runtime concept.
- Treat generated applications, sites, bots, dashboards, libraries, and other
  artifacts as task data or output, not as the tool's identity.
- Store product-specific choices in task payloads, manifests, fixtures,
  plugins, adapters, project-local configuration, service discovery, or
  user-selected state.
- Workflow/progress logs belong to the selected or active run. Detailed logs for
  completed runs should collapse by default or render as compact final status
  unless the user is debugging them.
- Build with SOLID where OOP exists and with equivalent module/contract
  boundaries in non-OOP stacks. Keep orchestration, product/domain logic, UI,
  persistence, filesystem, external services, and configuration in separate
  layers with explicit contracts.

## Suggested Audit

- Search touched runtime code and tests for demo/product names, customer names,
  fixed slugs, fixed stack selections, fixed ports/URLs, product-specific UI
  copy, and workflow steps tied to one task shape.
- Move scoped findings into task data, manifests, config, service discovery,
  fixtures, adapters, or example docs.
- Update tests so sample products prove generic behavior instead of becoming
  the default runtime contract.

## Verification

- Confirm runtime defaults use neutral names and labels.
- Confirm selected-run workflow logs are not global task boards and completed
  runs render compactly or collapsed by default.
- Run focused tests or smoke checks that cover one sample product/task while
  asserting generic runtime behavior.
- Confirm no secrets, private project data, production data, or local absolute
  paths were added while moving hard-coded values.

## Rollback

Remove only the added instruction text and references from project-local docs.
Do not reintroduce product-specific runtime defaults that were already replaced
with configuration, manifests, task payloads, or fixtures.
