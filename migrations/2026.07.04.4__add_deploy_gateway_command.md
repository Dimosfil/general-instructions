# Migration: Add Deploy Gateway Command

Migration ID: `2026.07.04.4__add_deploy_gateway_command`

## Purpose

Teach GI-enabled projects that `gi deploy <method-or-path>` / `ги деплой
<способ-или-путь>` means "deploy this current project through the explicitly
named method or deploy gateway", not "switch to that other project and work
there".

## Apply

- Update command indexes to include `gi deploy <method-or-path>` and Russian
  aliases.
- Update project operation runtime instructions so an absolute or project-local
  path after `gi deploy` or `gi ftp` is treated as a user-authorized deploy
  gateway path for the current command.
- Require agents to read the gateway's own `AGENTS.md`, `COMMANDS.md`, and
  deploy runbook before executing anything from the gateway.
- Require agents to use only the gateway's documented single entrypoint, passing
  the current project root as the source path and any project id, deploy mode, or
  target name only when the gateway contract defines those parameters.
- Preserve secret boundaries: do not print credentials, infer missing secrets,
  edit the gateway's private local config, or copy gateway runtime data into the
  source project.
- When the task is to prepare the deploy gateway itself, create or update the
  gateway-local contract files and entrypoint there; consuming projects should
  only call the documented gateway interface.

## Notes

Keep concrete local hub paths in project-local instructions or user messages.
Shared instructions should describe the reusable deploy-gateway contract without
hard-coding one machine's folder layout.
