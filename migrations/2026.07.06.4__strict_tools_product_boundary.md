# Strict Tools Product Boundary

Migration ID: `2026.07.06.4__strict_tools_product_boundary`

## Summary

Make the `tools/` boundary a strict classification rule. Tooling belongs under
`tools/`; product source, runtime packages, plugins, tests, full product docs,
and artifacts do not.

## Instructions

- Before creating or moving any file under `tools/`, classify whether it is
  tooling or product material.
- Allow `tools/` for durable development and agent tooling only: scripts,
  adapters, bootstrap commands, deployment helpers, verification helpers,
  agent-memory tooling, and small redacted examples or manifests.
- Do not put product runtime/source packages, product plugin implementations,
  product tests, full product documentation, generated product output,
  selected-run artifacts, uploaded site contents, screenshots, raw exports,
  build bundles, downloaded datasets, or one-off work results under `tools/`.
- Put product code under the project's source/package locations, tests under
  the test tree, product docs under `README.md`/`docs/`/runbooks, and artifacts
  under documented artifact, evidence, output, data, docs-asset, build, or
  release locations.
- Treat `tools/project-memory/` as a narrow exception for compact
  implementation-driving specifications, decisions, contracts, and evidence
  references. It is not a source package, plugin directory, product test tree,
  full documentation site, artifact bucket, or dump folder.
- If the requested write would violate this boundary and no project-local
  tooling contract explicitly allows it, stop and report the target-location
  blocker instead of silently writing into `tools/`.

## Rollback Or Conflict Notes

This rule does not ban scripts, adapters, bootstrap helpers, deployment helpers,
verification helpers, agent-memory tooling, redacted examples, small manifests,
or compact project-memory specifications from `tools/`. It bans using `tools/`
as a destination for product code, product tests, full product docs, and
artifacts.
