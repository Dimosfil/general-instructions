upstream, with cautious conflict handling.

## Apply

- Treat `gi пул`, `gi pull`, and `ги пул` as explicit requests to fetch and pull
  the current branch from its configured upstream.
- Before pulling, inspect `git status --short`, the current branch, and upstream
  configuration.
- Pull only the current branch; do not switch branches, rewrite history, rebase,
  or pull from another remote unless the user explicitly asks.
- If local changes make the pull unsafe, unresolved conflicts already exist, the
  project is not a git repository, or no upstream is configured, stop and explain
  the blocker.
- If conflicts appear during pull, inspect the conflicted files and resolve only
  obvious, low-risk conflicts where intent is clear and user changes are
  preserved.
- If a conflict requires product judgment, touches secrets or unrelated user
  changes, spans files outside the requested scope, or cannot be resolved with
  high confidence, stop and ask the user with a concise summary and options.

## Verify

- Ask `gi пул` in a project with a clean branch and upstream; confirm the agent
  fetches and pulls only the current branch.
- Ask `gi pull` in a project without upstream; confirm the agent stops and
  reports the missing upstream.
- Test or review a conflicted pull scenario; confirm only clear low-risk
  conflicts are resolved automatically and uncertain conflicts are escalated to
  the user.

## Rollback Or Conflict Notes

This migration adds a pull command only. It must not weaken the default rule
that agents do not run branch changes, history rewrites, destructive git
commands, commits, or pushes without explicit user request.
