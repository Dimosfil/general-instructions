# Classify Development Versus Refactor

Migration ID: `2026.07.01.1__classify_development_vs_refactor`

## Accepted Guidance

- Treat refactoring as structural work that preserves user-visible behavior and
  documented API, storage, workflow, and UI contracts.
- Treat new behavior, validation, observability, integrations, runtime flows,
  persistence shape changes, and tests that define new contracts as development
  work.
- Treat endpoint discovery, service lifecycle actions, smoke tasks, restarts,
  deploys, and release checks as verification or operational work, not code
  refactors.
- For mixed batches, name each part explicitly and apply the stricter rule set
  for behavior changes, public contracts, data, secrets, production systems, or
  external services.
- For pure mechanical moves, prove existing focused coverage before the move and
  rerun the same checks after the move. For behavior or contract changes, add or
  identify a focused guarding test when practical before implementing.
- When extracting or replacing modules, name the public contract: caller,
  callee, input shape, output shape, error and empty-state behavior, side-effect
  ownership, and invariants that must not leak across layers.
