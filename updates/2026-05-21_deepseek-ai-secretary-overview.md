# Краткий обзор: AI-секретарь на DeepSeek

Материал описывает экономную архитектуру AI-секретаря для почты, календаря и задач на DeepSeek. Главная идея — не отправлять каждый запрос сразу в полноценного агента, а сначала дешево классифицировать его и выбрать минимально достаточный путь обработки.

## Суть архитектуры

Система состоит из трёх уровней:

- Уровень 0: pre-checks без LLM — rate limit, длина сообщения, slash-команды.
- Уровень 1: router на `deepseek-chat` — короткая JSON-классификация интента.
- Уровень 2: выполнение — шаблонный ответ, `deepseek-chat` с инструментами или `deepseek-reasoner` для сложного анализа.

Роутер делит запросы на шесть категорий:

- `simple_action` — простые действия, которые можно выполнить одним инструментом.
- `chat` — короткий вопрос-ответ по данным.
- `compose` — составление письма или текста.
- `analyze` — анализ нескольких писем, задач или событий.
- `complex_reasoning` — многошаговое планирование и сложные решения.
- `out_of_scope` — всё вне почты, календаря и задач.

## Главные правила экономии

- Держать стабильный system prompt в начале, чтобы DeepSeek Context Caching давал cache hit.
- Выносить динамический контекст, время, письма и календарь в user message или ближе к концу запроса.
- Использовать `deepseek-chat` для большинства задач.
- Вызывать `deepseek-reasoner` только для действительно сложного анализа и только после предварительного сбора данных.
- Всегда задавать `max_tokens` явно.
- Отвечать шаблонами без LLM для подтверждений, лимитов и off-topic.
- Использовать RAG вместо помещения всей переписки в контекст.

## Особенности моделей

`deepseek-chat` подходит для роутинга, быстрых ответов, function calling, JSON mode и обычной агентской работы с инструментами.

`deepseek-reasoner` подходит для сложных рассуждений, но не поддерживает tools и JSON mode напрямую. Поэтому данные для него нужно собрать заранее через обычную chat-модель и инструменты.

## Безопасность и контроль

Материал предлагает ограничивать:

- длину пользовательского сообщения;
- число сообщений истории;
- размер писем в контексте;
- число писем в batch-контексте;
- запросы в минуту и день;
- дневной токен-бюджет пользователя;
- число tool-calls за один ход.

Рискованные действия требуют подтверждения пользователя: отправка писем, удаление, массовые операции и изменение событий с участниками. Текст писем, задач и событий нужно считать данными, а не инструкциями, чтобы защититься от prompt injection.

## Практический вывод

Это хороший reusable-паттерн для почтового AI-ассистента: он снижает стоимость, сохраняет управляемость, отделяет быстрые действия от сложных рассуждений и задаёт понятные границы для инструментов, RAG, лимитов и подтверждений пользователя.
